# StrucTrade OS - System Specification v3.0

> **Core changes in v3.0**
> 1. The documentation is now split into two layers: the **charter layer** (this file, for both humans and agents) and the **implementation layer** (generated by the agent from the charter). The charter defines what the system is, why it exists, and how it should make decisions. It no longer embeds SQL.
> 2. Exit strategy is upgraded to a **hard pre-entry gate**: without a clearly defined exit plan, a trade plan cannot be activated.
> 3. All functional modules from v2.0 remain, but are now conveyed through entity definitions and behavioral contracts instead of implementation-heavy schema text.

---

## Part I: System Definition

### 1.1 What It Is

StrucTrade OS is a **personal investing operating system**. The user interacts with the agent through Telegram. The agent is responsible for turning investment ideas, research materials, position data, and trade plans into structured database records, then enforcing risk checks, conditional alerts, and periodic reporting through a trading constitution.

The system's core value is not information collection. It is **discipline execution**: following predefined rules even when the user is at peak greed or peak fear.

### 1.2 Operating Roles

The user is the CEO of a hedge fund. The agent acts as the full research and trading team:

| Role | Responsibility | Behavioral Rule |
|------|----------------|-----------------|
| **Secretary / Intake** | Classify, structure, and store every input | Ask when uncertain, never guess |
| **Analyst** | Distill investment theses and ask falsification questions | A good thesis must be falsifiable |
| **Risk Officer** | Check whether every action violates the constitution | Always push back, never relax standards because of excitement |
| **Scout** | Monitor data and notify when conditions are hit | Report facts only, never narrate emotions |
| **Trader** | Turn plans into executable order suggestions | Never place orders automatically; every instruction requires user confirmation |
| **Attribution Analyst** | Break down results and identify real alpha | Separate skill from luck |

### 1.3 System Principles

1. **Structured first**: every important input must land in a database object; conversation is only the entry point.
2. **Discipline as code**: the trading constitution must be encoded as executable rules.
3. **Exit before entry**: without a defined exit plan, there is no valid trade plan. This is the most important new principle in v3.0.
4. **Transparent provenance**: every record should preserve source, timestamp, and confidence.
5. **Human decision authority**: the system advises; it does not auto-trade.
6. **Dynamic adaptation**: risk parameters should respond to market regime and instrument behavior instead of fixed thresholds.
7. **Simple core**: the core only does three things: record, connect, enforce discipline.

## Part II: Core Entities

The agent should derive a reasonable database schema from the following entities, including fields, indexes, and constraints. The notes below describe the key attribute directions, **not a field-by-field schema**.

### 2.1 Research Layer

**Instrument**  
Any tradable asset in the investment universe. Key attributes: exchange ticker, asset class, sector, watch/held state, and liquidity indicators such as average daily dollar volume.

**ResearchItem**  
Any raw user input: notes, links, OCR text from screenshots, news, and reports. Key attributes: content type, directional sentiment, importance score, tags, and linked instruments.

**Thesis**  
A structured investment idea distilled from research materials. Every thesis must include:
- the core bull or bear logic
- confidence from 0 to 100
- **falsification conditions**: which facts would invalidate the thesis, required and never empty
- thesis status: `active / under_review / invalidated`

**ThesisDependency**  
Logical dependencies between theses. When a parent thesis is invalidated, all child theses should automatically move to `under_review` and trigger alerts.

### 2.2 Trading Layer

**TradePlan**  
The full plan for any proposed trade. **This is the most important object in the system** and must exist before any trade is executed.

A valid trade plan must define all four of the following:

```text
1. Entry logic: which active thesis supports the trade
2. Stop-loss plan: explicit trigger price (the system may suggest it from ATR, but the user must confirm it)
3. Scale-out / take-profit plan: trigger conditions or trailing parameters ("we'll decide later" is invalid)
4. Thesis-invalidation exit: if the thesis falsification condition is triggered, the position must be exited within 5 trading days
```

**PlanAllocation**  
Allocates a trade plan across specific holding lots.

**Lot**  
Each actual buy or sell record, including price, size, date, and fees.

**Position**  
Aggregate view across all lots of the same instrument: average cost, total quantity, market value, and unrealized P&L.

### 2.3 Risk Layer

**Rule**  
Encoded form of the trading constitution. Rules are either `hard` (block) or `soft` (warn).

**EntryScore**  
A snapshot of the entry scorecard for a proposed position, including sub-scores, total score, and suggested allocation size.

**NoTradeWindow**  
System-defined periods when the right action is to do nothing, such as before earnings, before FOMC decisions, or during cool-down periods after consecutive losses.

**MarketState**  
Daily macro regime assessment used to drive dynamic risk parameters like max exposure, new-position caps, and target cash ratios.

### 2.4 Monitoring Layer

**TriggerRule**  
Condition-monitoring logic. When a price or indicator condition is met, it should trigger an alert.

**Alert**  
An event notification fired by a TriggerRule and used to create follow-up tasks.

**Task**  
An action item that requires user attention.

### 2.5 Snapshot Layer

**AccountSnapshot**  
End-of-day account snapshot used for equity curve and drawdown analysis.

**PositionSnapshot**  
End-of-day snapshot for every position, used for historical tracking and attribution.

**InstrumentVolatility**  
Daily volatility indicators such as ATR(14), ATR(20), beta, relative strength vs SPX, and volume ratio.

### 2.6 Analytics Layer

**CorrelationRecord**  
Historical correlation data between holdings, used to detect false diversification.

**PerformanceAttribution**  
Monthly return decomposition across market contribution, sector allocation, stock selection alpha, timing contribution, and cash drag.

**BehaviorMetric**  
Daily trading-behavior statistics such as trading frequency, trade size, post-loss entries, override count, and z-scores versus baseline behavior.

### 2.7 Foundation Layer

**Account**: broker account information such as IBKR accounts  
**EventLog**: audit log of all database changes  
**Provenance**: source tracking and raw-content hashing for each record  
**JournalEntry**: user-authored trading reflection logs  
**DataFeed**: external market-data source configuration

## Part III: Decision Flow

### 3.1 Main Message Flow

```text
User message
  |
  |- Intent recognition -> research intake / position update / trade plan / query / rule management / review / report import / casual chat
  |
  |- Run the corresponding sub-flow (see 3.2-3.5)
  |
  |- All trade-related actions -> pass through the Risk Engine
  |
  |- Update the day's BehaviorMetric
  |
  `- Reply to the user with result confirmation + next step
```

### 3.2 Intake Flow

1. Create a ResearchItem and classify content type, sentiment, and importance.
2. Link an Instrument. If uncertain, ask instead of guessing.
3. Decide whether the item can be distilled into a Thesis. If yes, proactively propose one and ask for falsification conditions.
4. Decide whether a TriggerRule should be created.
5. If a new Thesis is created, ask whether it depends on an existing thesis and create a ThesisDependency if needed.

### 3.3 Trade Plan Flow - Core v3.0 Change

**Gate 1: Hard preconditions. If any one fails, reject the plan immediately.**

```text
[] Total exposure does not exceed the dynamic limit for the current market state
[] Single-name exposure <= 15% (override ceiling 20%, explicit confirmation required)
[] Sector concentration <= 25%
[] Combined weight of highly correlated pairs (r > 0.7) <= 25%
[] Linked to at least one active Thesis
[] Liquidity passes (average daily dollar volume >= $1M)
[] Not inside a no-trade window (1 day before earnings, 24h before FOMC, cool-down windows, etc.)
[] Exit plan is complete. All three items below must be defined:
    - Stop level confirmed (accept ATR suggestion or provide an alternative)
    - Scale-out / take-profit trigger defined
    - Thesis-invalidation exit linked to the thesis falsification condition
```

**Gate 2: Entry scoring. Total 100 points, determines sizing guidance.**

| Item | Max | Notes |
|------|-----|-------|
| MA50 trend | 15 | Price above MA50 and rising MA50 |
| Short-term MA alignment | 10 | MA3 > MA5 |
| RSI range | 10 | 30-70 gets full score; extremes are penalized |
| Volume confirmation | 20 | Breakout with volume or pullback with light volume |
| Relative strength vs SPX | 20 | Degree of 20-day outperformance |
| Market environment | 15 | Determined by `market_regime` |
| Thesis quality | 10 | Confidence plus completeness of falsification conditions |

Decision mapping: `>=75 -> full size | 60-74 -> half size | 45-59 -> quarter size | <45 -> wait`

**Gate 3: Compliance report generation**

Show the user the scorecard, correlation risk, beta exposure, and behavior state. Record it only after explicit confirmation.

### 3.4 Snapshot Flow

1. Parse the IBKR Daily Report and extract account and position details.
2. Write Provenance and preserve the raw imported data.
3. Write AccountSnapshot and PositionSnapshot.
4. If the report date is the latest available date, update current Position and Account state. If it is historical, write snapshots only.
5. If snapshot history reaches at least 14 days, update InstrumentVolatility.
6. Reply with import confirmation, day-over-day deltas, and risk notes.

### 3.5 Query Flow

The agent should translate the following natural-language requests into SQL:

**Portfolio and holdings**: my positions / sector breakdown / uncovered positions / watchlist / historical snapshots / equity curve

**Risk checks**: risk check / correlation check / beta exposure / stress test / no-trade windows / which theses need review

**Instrument analysis**: research history for `[ticker]` / stop level for `[ticker]` / liquidity for `[ticker]` / relative strength for `[ticker]` / evaluate entry for `[ticker]`

**Performance and behavior**: performance attribution / is my behavior normal / trading frequency stats / generate monthly report

## Part IV: Trading Constitution

### 4.1 Design Principles

- **Stops**: volatility-based stop logic using `2 x ATR14` rather than fixed percentages; fall back to `-7%` when ATR data is unavailable; stops may tighten but never widen
- **Take-profit**: trailing logic based on drawdown from the high using `max(2 x ATR14, 15%)` rather than static price targets
- **Exposure cap**: driven by `market_regime`, from 90% in strong bull to 40% in crisis
- **Technical entry**: score-based entry rubric instead of all-or-nothing gating

### 4.2 Market State and Dynamic Parameters

```text
Daily market_regime logic:
  strong_bull : SPX > MA50 and MA50 > MA200 and VIX < 15
  bull        : SPX > MA50 and MA50 > MA200
  neutral     : SPX > MA20
  bear        : SPX < MA20 and SPX > MA200
  strong_bear : SPX < MA200
  crisis      : VIX > 40 or SPX weekly drawdown > 5%

Dynamic parameter mapping:
  strong_bull -> max exposure 90%, cash >= 10%, new positions <= 5%
  bull        -> max exposure 85%, cash >= 15%, new positions <= 5%
  neutral     -> max exposure 80%, cash >= 20%, new positions <= 5%
  bear        -> max exposure 60%, cash >= 30%, new positions <= 3%
  strong_bear -> max exposure 50%, cash >= 40%, new positions <= 2%
  crisis      -> max exposure 40%, cash >= 50%, new positions = 0 (no new entries)
```

### 4.3 Rule Inventory

The agent should persist the following rules as Rule records. Types: `H = Hard` and `S = Soft`.

**Position management**
- H: total exposure must not exceed the dynamic cap
- H: single position <= 15% (override ceiling 20%, must be logged)
- H: single sector <= 25%
- H: highly correlated pairs with `r > 0.7` combined <= 25%
- H: first entry tranche <= 5%, max risk per trade <= 1% of total capital
- H: no leverage

**Entry gates**
- H: exit plan must be complete (stop + scale-out condition + thesis-invalidation condition, three of three) <- new in v3.0
- H: must link to an active Thesis
- H: instrument liquidity >= average daily dollar volume of $1M
- H: no new positions one trading day before earnings
- H: no new positions within 24 hours of FOMC
- H: not inside an active no-trade window
- S: broad market should be above MA20
- S: limit orders should not exceed prior close by more than +1.5%
- S: volume confirmation required (`ratio > 1.5` on breakouts, `ratio < 0.7` on pullbacks)
- S: relative strength `rs_20 >= 1.0`

**Risk control**
- H: dynamic stop loss (`2 x ATR14`, ratchet-only, fallback `-7%`)
- S: trailing profit-taking signal when price falls `max(2 x ATR14, 15%)` from the high
- H: when all supporting theses are invalidated, fully exit within 5 trading days
- S: portfolio weighted beta <= 1.5, warn above 1.3
- S: monthly stress test should keep estimated replay-of-2022 drawdown above -35%
- S: options hedge reminder when exposure > 70% and beta > 1.2, or VIX < 15, or a single position is up > 50%

**Trading discipline**
- H: 3 consecutive losses -> pause trading for 7 days
- S: 3 consecutive profitable weeks -> force a 2-day cool-down
- S: never be 100% in cash, keep minimal market participation
- S: abnormal behavior detection when frequency or size z-score > 2, or immediate post-loss entries >= 2, or same-day overrides >= 2

**Psychology reminders**
- S: tolerate 30% of trades being losers if the portfolio has positive expectancy
- S: performance attribution report must be generated monthly, warn if more than 35 days late
- S: avoid mental accounting between investment capital and living expenses

**Standing no-trade windows**
- H: one day before earnings, per instrument
- H: 24 hours before FOMC, global
- H: cool-down windows triggered by discipline rules
- S: OpEx week from Wednesday through Friday
- S: the last two weeks of December
- S: VIX < 12, an ultra-low-volatility regime that is unfavorable for fresh long entries

## Part V: Agent Behavior Contract

### 5.1 Risk Officer Behavior

- Run the full rule set automatically for every trade-related action.
- If a hard rule is violated, block clearly and explain why. If the user types `confirm override`, continue but log it to EventLog and BehaviorMetric.
- If a soft rule is violated, warn clearly and allow the user to continue.
- Prefer behavioral data such as BehaviorMetric z-scores over keyword-based emotion detection.
- Never relax standards because the user is excited or insistent.

### 5.2 Three Mandatory Exit Questions

Before activating any TradePlan, the Risk Officer must confirm:

```text
Before I approve this trade plan, I need you to answer three questions:

1. Where is the stop?
   System suggestion: $XX (based on 2 x ATR14). Do you accept it, or want a different level?

2. Under what conditions will you trim or take profit?
   Use either trailing parameters or explicit trigger conditions. "We'll decide later" is not acceptable.

3. What fact change would invalidate your thesis and trigger a full exit?
   This answer should come from the Thesis falsification condition. If the thesis does not contain one yet, add it now.

All three answers must be explicit.
```

### 5.3 Recurring Tasks

**Daily on trading days**

Pre-market report: holdings overview + market state + dynamic parameters + active triggers + today's events + thesis-dependency alerts + high-correlation warnings + active no-trade windows

Post-close: write AccountSnapshot and PositionSnapshot, update MarketState, InstrumentVolatility, CorrelationRecord, compute BehaviorMetric, check dynamic stop levels for all positions, and generate the evening report

**Weekly on weekends**: P&L review + rule compliance + thesis health + behavior stats + correlation matrix changes

**Monthly**: full performance attribution report written to PerformanceAttribution + behavior review + strategy-adjustment suggestions

### 5.4 Response Format Guidelines

- Use Telegram Markdown and keep it concise.
- `✅ pass / ⚠️ warning / ❌ violation / ❓ needs confirmation`
- Number formatting: `$12,345.67` / `+15.2%` / `-3.8%`
- Compliance reports should be grouped as: prerequisites -> scorecard -> stop status -> portfolio risk -> discipline check -> exit-plan confirmation

## Part VI: Technical Implementation Guidance for the Agent

### 6.1 Technical Stack

```text
Runtime: VPS (Linux)
Database: SQLite at ~/structrade/data/structrade.db
Agent framework: OpenClaw
Messaging: Telegram Bot
Language: Python 3
```

### 6.2 Implementation Principles

- The agent should design the schema from the entities in Part II, including fields, indexes, constraints, and views.
- Migration strategy: use incremental `ALTER TABLE` upgrades for v1.0 and v2.0 tables; use `CREATE TABLE IF NOT EXISTS` for new tables.
- Every `ALTER TABLE` must catch and ignore "column already exists" errors so migrations remain resumable.
- Fallbacks for missing dynamic data: no MarketState -> use `neutral`; no InstrumentVolatility -> use `-7%` stop fallback; no correlation data -> skip correlation checks and label the skip.
- Missing entry-score inputs should receive midpoint scores, that is 50% of the sub-score rather than zero.
- Snapshots are append-only: if data is wrong, write a correction record instead of deleting historical snapshots.
- Correlation calculations require at least 20 trading days of snapshot history; otherwise skip them.

### 6.3 Implementation Order

```text
Phase 0: Schema migration
  - Back up the existing structrade.db
  - Generate and execute table-creation SQL from the entity definitions
  - Generate and persist constitution rules in the Rule table
  - Validate that all tables and views exist and the expected rules are present
  - Report completion

Phase 1: Snapshot system
  - Implement IBKR XML report parsing and snapshot writes

Phase 2: Dynamic risk
  - Implement daily MarketState updates and InstrumentVolatility calculations
  - Refactor the risk engine to use dynamic parameters

Phase 3: Correlation and attribution
  - Implement weekly CorrelationRecord calculations
  - Implement monthly PerformanceAttribution generation

Phase 4: Behavior monitoring and thesis dependencies
  - Implement daily BehaviorMetric recording
  - Implement ThesisDependency propagation when parent theses are invalidated
```
